## 1. TEMEL FELSEFE (Değişmez 5 Kural)

Bu 5 kural sözleşme niteliğindedir, değiştirilemez:

### Kural 1 — Personel = Tek Omurga
Tüm sistemler (yetki, görev, üretim, maliyet, vardiya) `personel_kullanici.id` üzerinden bağlanır. Tek omurga, parçalı sistem yok.

### Kural 2 — Personel ≠ Login
Sisteme login = `sistem_kullanici`. Personel olmak için login şart değildir. Bir operatör sahada çalışır, üretim kaydı yapılır, prim alır — ama hiç sisteme login olmadan. Bu bilinçli bir ayrımdır.

### Kural 3 — CPS Master, Dış Sistem Kaynak
PDKS, 5055, Korgun — hepsi **veri kaynağıdır**. CPS, **otorite** sahibidir. Veri akış yönü: dış sistem → CPS enrichment. Override CPS'te.

### Kural 4 — Veri Zincirleri Kaybedilmez
`legacy_id`, üretim geçmişi, audit log, görev geçmişi, maaş audit, vardiya değişimi — hiçbiri silinmez. Soft delete (Aktif=0) zorunludur.

### Kural 5 — Davranış Değişimi Disipline
Her sprint sıfır davranış değişimi hedefler. Migration sırasında saha personeli üretim yapmaya devam edebilmelidir. Davranış değiştiren değişiklikler ayrı sprint, ayrı onay, ayrı snapshot ile yapılır.

## 2. SOIS KATMAN MİMARİSİ

Solariz organizasyonu **3 katmanda** modellenir:

```
KATMAN A: DEPARTMAN (organizasyonel)
  ├─ Üretim
  ├─ Planlama
  ├─ Kalite
  ├─ Finans
  ├─ Grafik
  ├─ İdari
  ├─ Yönetim
  └─ Çin Ofis

KATMAN B: EKİP (uzmanlık grubu)
  Üretim → [Enjeksiyon, Silikon, Paketleme, Kalıphane]
  Planlama → [Operasyon Planlama, Sipariş Yönetimi]
  Kalite → [AQL, Lisans, Numune]
  Finans → [Muhasebe, Cari, İthalat]
  Grafik → [Tasarım, Baskı]
  İdari → [İK, Genel]
  Yönetim → [Üst Yönetim]
  Çin Ofis → [İhracat]

KATMAN C: PERSONEL (gerçek insan)
  - personel_kullanici.id (canonical PersonelId)
```

### Şirket Katmanı

Şu an Solariz tek şirket. Şirket katmanı **atlanır**. İleride çoklu şirket olursa ayrı sürüm.

### Departman ≠ Rol (Kritik Ayrım)

> **Karar 7 (Ek Kritik Kararlar):** *Departman, yetki sistemi ile birebir bağlanmasın.*

| Departman | Rol (sistem_rol) |
|---|---|
| **Organizasyonel kimlik** | **Yetki seti** |
| "Bu kişi nerede çalışır?" | "Bu kişi ne yapabilir?" |
| Statik, az değişir | Dinamik, sık güncellenir |
| Üretim, Finans, ... | Yönetim, Enjeksiyon, Planlama, ... |

Bir personel **bir departmanda** olur, **birden fazla rolü** olabilir. Departman organizasyondur, rol yetkilendirmedir.

## 6. YILDIZ SİSTEMİ

### 6.1 Felsefe

> *"Yıldız sistemi çok doğru fikir. Ama ceza sistemine dönüşmemeli."*

Yıldız bir **takdir aracı**dır. Düşük yıldız = "daha çok desteğe ihtiyacı var" mesajı, "kötü çalışan" damgası değil. Bu felsefe UI'da, raporlarda, AI önerilerinde **dil seçimi** olarak yansıtılır.

**Yasak ifadeler:**
- ❌ "Düşük performanslı personel"
- ❌ "Cezalandırılması gereken"
- ❌ "Hedef altı kişi" (negatif odak)

**Olumlu ifadeler:**
- ✓ "Destek bekleyen"
- ✓ "Gelişim alanı olan"
- ✓ "Yüksek potansiyelli"

### 6.2 Hibrit Hesaplama

```
YildizSon = (ManuelYildiz × Agirlik_M) + (OtomatikYildiz × Agirlik_O)

Default ağırlıklar:
  Agirlik_M = 0.4  (manuel %40)
  Agirlik_O = 0.6  (otomatik %60)

OtomatikYildiz hesabı (her bileşen 1-5 arası):
  Y_uretim    = uretim_verim_oran * 5         # hedefin yüzdesi
  Y_devamsiz  = 5 - (gec_kalma_gun / 10)
  Y_kalite    = 5 - (red_orani * 5)
  Y_gorev     = gorev_tamamlama_orani * 5

  OtomatikYildiz = ortalama(Y_uretim, Y_devamsiz, Y_kalite, Y_gorev)
```

**Hesaplama frekansı:** Haftalık (her Pazartesi cron).

### 6.3 Manuel Yıldız Boyutları

Manuel kısım sadece tek sayı değil, **çoklu boyut** olabilir:

  - Öğrenme isteği (1-5)

## 7. EKİP LİDERİ

### 7.1 Tek Seviye Kararı

> *"Ekip lideri mantığı çok doğru. Ama recursive kaos istemiyorum. Şimdilik: tek lider yeterli. Organizasyon ağacı sonra."*

Mimaride **iki ayrı kavram**:

| Kavram | Tablo | Açıklama |
|---|---|---|
| **Doğrudan lider** | `personel_kullanici.EkipLideriId` | Kişiye 1:1 bağlı lider |
| **Ekip lideri** | `ekip.LiderPersonelId` | Bir ekibin başındaki kişi |

Bu ayrım kasten korunur:
- Ahmet operatör → `EkipLideriId = Ferhat` (kişisel lider)
- Ferhat → `ekip.LiderPersonelId` (Enjeksiyon Ekibi lideri)

### 7.2 Soft Limit

Ferhat'ın iki rolü:

- Ferhat ekibinden ayrılınca lider değişir, çalışanların kişisel lideri değişmez (yeni atama gerek)
- Çalışan başka ekibe geçince lideri kalabilir (mentor ilişkisi)

## 8. MAAŞ SİSTEMİ

### 8.1 Tarihsel Tablo Zorunlu

> *"Maaş: ayrı tarihsel tablo olmalı. Asla tek kolon overwrite mantığına dönme."*

`personel_kullanici.Maas` kolonu **şu anki güncel maaş** için. Geçmiş kayıt **personel_maas** tablosunda.

### 8.2 Tablo Tasarımı

```sql
CREATE TABLE personel_maas (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    PersonelId INTEGER NOT NULL,

    -- Maaş bilgisi
    Tutar NUMERIC NOT NULL,
    ParaBirimi TEXT DEFAULT 'TL',           -- TL, USD, EUR
    Tip TEXT,                                -- 'maas', 'prim', 'avans', 'mesai', 'tazminat'

    -- Zaman aralığı
    BaslamaTarih TEXT NOT NULL,
    BitirmeTarih TEXT,                       -- NULL = aktif

    -- Audit
    KimAtadi INTEGER,                        -- PersonelId of admin
    Aciklama TEXT,
    OlusturmaTarih TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (PersonelId) REFERENCES personel_kullanici(id),
    FOREIGN KEY (KimAtadi) REFERENCES personel_kullanici(id)
);

CREATE INDEX idx_maas_personel ON personel_maas(PersonelId, BaslamaTarih DESC);
```

### 8.3 Güncel Maaş Hesabı (View)

```sql
CREATE VIEW v_personel_guncel_maas AS
SELECT
    PersonelId,
    Tutar,
    ParaBirimi,
    BaslamaTarih
FROM personel_maas
WHERE BitirmeTarih IS NULL
  AND Tip = 'maas';
```

### 8.4 Maaş Güncelleme Süreci

```
Admin "Maaş güncelle" butonu:
  1. Mevcut aktif maaş kaydının BitirmeTarih = bugün
  2. Yeni kayıt INSERT (Tutar, BaslamaTarih=bugün, BitirmeTarih=NULL)
  3. personel_kullanici.Maas = yeni_tutar (cache, hızlı erişim için)
  4. personel_kullanici.MaasGuncellemeTarih = bugün

```

### 9.2 Merge Algoritması (SINIF 1)

```python
def merge_personel(eski_pid, yeni_pid, onaylayan_id):
    """
    eski_pid'in tüm üretim kayıtlarını yeni_pid'e taşı.
    Audit edilir.
    """
    BEGIN TRANSACTION:
        # 1. Audit kayıt
        INSERT personel_identity_map (
            PersonelId=yeni_pid,
            KaynakSistem='manuel',
            LegacyId=eski_pid,
            EslestirmeYontemi='manuel_merge',
            GuvenSkoru=100,
            OnaylayanPersonelId=onaylayan_id,
            Notlar='Merge: ' + str(eski_pid) + ' -> ' + str(yeni_pid)
        )

        # 2. Üretim kayıtları taşı
        UPDATE uretim_kayit SET personel_id = yeni_pid
        WHERE personel_id = eski_pid

        # 3. Diğer ilişkiler (tasks, vb.) taşı

        # 4. Eski PK kaydı (varsa) PASIF
        UPDATE personel_kullanici SET Aktif=0
        WHERE id = eski_pid
    COMMIT
```
